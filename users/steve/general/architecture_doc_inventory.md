## What this inventory tells us

**There is no canonical architecture document.** There is:

1. A **front-door README** with pointers and package names.
2. A **principles doc** (`BRIDGES.md`) that names one
   structural idea but not the system.
3. A **glossary** (`GLOSSARY.md`) that names the vocabulary.
4. A **per-flow walk-through** (`USER_FLOWS.md`) scoped to
   the Elm client only.
5. A **generated index** of sidecars (`LABELS.md`).
6. ~100+ **sidecars** (`*.claude`), one per module, describing
   local role.
7. ~50 **project memories** capturing discrete architectural
   atoms.
8. A **growing essay pipeline** with per-topic depth dives,
   often the first place a new principle crystallizes.

Nothing unifies them. For a new contributor (or future-me),
the path to "how does Angry Gopher work as a system" is: read
README → follow pointers → read 5–10 sidecars in relevant
subtrees → search memories → read ~3 relevant essays →
reconstruct the graph in your head.

That's the gap a real architecture doc would close.

## Suggested shape for the real doc

Based on what exists and what's missing, a minimal first
draft probably wants these sections:

1. **Mission** — the agent-plays-watchable-in-Elm product.
2. **The big picture diagram** — Go server at the middle,
   Elm UI on one side, Python agent on the other, SQLite
   below, all talking one action-log contract.
3. **Core data model** — action log + seed; the durable vs.
   rich split (table from `faithful_vs_durable_capture`);
   frames of reference (board vs. viewport).
4. **Component roles** — what each of Go/Elm/Python/DB is
   responsible for; who knows what; who decides what.
5. **Cross-language contracts** — the wire format (primitives,
   envelope shape, gesture_metadata), DSL conformance
   (shared fixtures), provenance.
6. **Design principles** — redundancy-as-asset (link to
   BRIDGES.md), record-facts-decide-later,
   own-the-whole-system, hints-are-client-side.
7. **Where to find more** — point at sidecars, LABELS,
   memories, essays.

Sections 3 and 4 are the highest-value because nothing today
says "here's the data model" or "Go does these things, Elm
does these things, Python does these things" in one place.
